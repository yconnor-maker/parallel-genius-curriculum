The API Key in the Public Repo

An AI agent wrote working code. It also committed a Personal Access Token to a public GitHub repo. Anyone who saw it could read and write the entire database.

Referenced in: M02, M04 Source: PHN Command Center, Apr 27 Severity: Critical

What Happened

An AI agent was tasked with building a command center dashboard that pulled data from Airtable. The agent needed an API token to authenticate. It did what any fast-moving developer does: it hardcoded the token directly in the JavaScript source file.

That file was committed to a public GitHub repository.

The Exposure

The Personal Access Token patywiQ5r7v6g2QgC... was committed in plain text inside phn-core.js. This token had full read/write access to the Airtable base applGxe8BtUlTwdpF — containing the entire operational database. Until revoked, anyone who viewed the repository could read, modify, or delete any record.

The Timeline

Agent builds

Code is generated

AI agent creates a working Airtable-connected dashboard. The token is placed directly in the fetch headers — Authorization: Bearer patywiQ5r7... — because that's the fastest path to a working demo.

git push

Token ships to GitHub

The code is committed and pushed to a public repository. The token is now in the git history — even deleting the file won't remove it from previous commits.

Discovery

Human review catches it

A subsequent session reviewing the codebase identifies the hardcoded credential. The finding triggers immediate remediation.

Remediation

4-step fix

PR #4 removes the token from source. A Netlify serverless proxy function replaces direct API calls. The leaked token is revoked in Airtable. A new, scoped token is issued with minimal permissions.

Why AI Agents Create This Problem

Speed Over Security

AI agents optimize for "make it work." The fastest path to a working API integration is a hardcoded token. The agent doesn't instinctively think about secrets rotation, environment variables, or credential scoping — it thinks about producing a functional result.

No Sense of "Public"

An AI agent doesn't distinguish between a private scratch file and a public repository. It writes the same code regardless. The concept of "this will be visible to the internet" doesn't factor into its code generation.

Compound Velocity

In an AI-assisted workflow, code goes from conception to deployment in hours, not days. There's less time for human review. The security review step that a traditional PR process enforces gets compressed or skipped entirely.

The Fix: Credential Hygiene as Standard Operating Procedure

1. Never Hardcode — Use Environment Variables

All API tokens go in environment variables (.env files locally, platform-level env vars in production). The codebase references process.env.AIRTABLE_TOKEN, never the actual token string.

2. Use a Serverless Proxy

Client-side JavaScript should never contain API credentials. A serverless function (Netlify Functions, Vercel Edge, etc.) sits between the browser and the API. The browser calls your function; the function calls the API with the token stored server-side.

3. Scope Your Tokens

Never use a full-access token. Create tokens with the minimum permissions needed — read-only if the app only reads, specific-table access if it only needs one table. A leaked read-only token is a problem; a leaked read-write token is a disaster.

4. Add to .gitignore Immediately

The .env file goes in .gitignore before the first commit. Not after. Before. This must be part of the project initialization checklist, not an afterthought.

5. Include in Agent Constitution

If you're using AI agents to write code, your Agent Constitution (the rules document every agent reads) must include: "Never hardcode API credentials. All secrets go in environment variables." The agent won't know unless you tell it.

"AI agents don't instinctively think about secrets rotation — they optimize for 'make it work.' Credential hygiene must be part of the standard operating procedure, not a post-incident discovery."
