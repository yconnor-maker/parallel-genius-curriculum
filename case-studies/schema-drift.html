Infrastructure Case Study

Schema Drift

The code said Timeline Check. The database said Timeline/Continuity Check. A field renamed in the UI broke every downstream agent that referenced it.

Referenced in: M06, M08 Source: PHN Command Center Fix, Apr 27 Pattern: Rename tolerance

What Happened

The PHN Command Center dashboard was built to pull QC (quality control) data from an Airtable base and display completion status for each scene in a fiction catalog. The code referenced specific field names to calculate progress.

Weeks later, a human — working through the Airtable UI — renamed several fields to be more descriptive. The UI updated. The database updated. The code did not.

Why This Happens in AI-Assisted Systems

Two Interfaces, No Sync

Humans rename fields through a GUI. Agents read field names from code. There's no mechanism to notify the code that the database changed. The rename is invisible to the agent.

Optimistic Assumptions

The original agent assumed the field names were stable. It hardcoded them as string constants. Nobody told the next agent that the field names had changed — because nobody told anyone.

Ghost References

Some fields were planned but never created. The code referenced Gold Lines Preserved — a field that existed in the design doc but was never added to the database. The code compiled. The feature silently failed.

The Fix: Build for Rename Tolerance

1. Use Field IDs, Not Names

Airtable, Coda, and most databases have stable internal IDs (fldXXXXXX). Reference those in code. Let the human rename the display label freely.

2. Schema Validation on Startup

When the dashboard loads, fetch the current field list and compare against expected fields. If a field is missing, show an error — don't silently skip it.

3. Fuzzy Matching Fallback

If Timeline Check isn't found, search for fields containing "timeline" and "check". This handles the common case where a field was renamed with additional descriptors.

4. Agent Constitution Rule

Add to your agent's rules: "After any session that modifies database schema, update the corresponding code constants and document the change in the schema changelog."

"In a system where humans rename fields through the Airtable UI and agents read field names from code, drift is inevitable. Build for rename tolerance."
