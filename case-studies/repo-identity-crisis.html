The Repo Identity Crisis

Two repositories. Same purpose. Different names. One gets built out, one doesn't. Six months later, nobody remembers which is canonical.

Referenced in: M04, M08 Source: YC Ops Hub Refactor, Apr 2026 Pattern: Naming as infrastructure

What Happened

An AI agent created a repository called yc-ops-hub for the operational dashboard. A different session — different agent, different day — created labora-ops-hub for the same purpose with slightly different content.

Both repos accumulated commits. Both had READMEs. Both were linked from different documents. When the human came back to consolidate, neither was clearly canonical. Decisions had been split across both.

❌ Deprecated

labora-ops-hub

Created by Agent A on April 12. 14 commits. Linked from Coda task board. Contains early dashboard prototypes and some live pages.

✅ Canonical

yc-ops-hub

Created by Agent B on April 15. 47 commits. Linked from curriculum syllabus. Contains production deployment, all current pages, and the Netlify config.

The human had to manually trace every link, every Netlify deploy, and every reference to determine which was the living system. Then migrate the 3 useful pages from the dead repo into the live one.

Why This Happens in Multi-Agent Systems

No Shared Registry

Each agent session operates independently. Agent B didn't know Agent A had already created a repo for this purpose. There was no central registry of "repos that exist and what they're for."

Naming Isn't Trivial

yc-ops-hub vs labora-ops-hub — both are reasonable names. But when you have 12 repos and 4 agents, naming becomes infrastructure. A naming convention prevents collision; ad hoc naming guarantees it.

Proliferation Is the Default

AI agents default to creating new things. They rarely check whether the thing already exists. Without an explicit instruction to "check first," every new session is an opportunity for duplication.

The Fix: Naming as Infrastructure

1. Naming Convention in the Agent Constitution

Define the pattern: {org}-{project}-{function}. All agents read this before creating anything. If the agent needs a new repo, it checks the existing list first.

2. Registry Before Creation

Maintain a living document (in Coda, Airtable, or a markdown file) that lists every repo, its purpose, and its canonical status. Before creating a new repo, the agent must check the registry.

3. Archive, Don't Delete

When you identify a duplicate, archive the dead repo (rename it with a DEPRECATED- prefix) rather than deleting it. Some decisions may live only in its commit history.

4. One Source of Truth Rule

For any given function (curriculum, ops hub, fiction production), there is exactly ONE canonical repository. This is documented, linked, and enforced. Duplicates are merged within 48 hours.

"In a system where agents create repos freely, naming conventions aren't style guides — they're collision prevention infrastructure."
